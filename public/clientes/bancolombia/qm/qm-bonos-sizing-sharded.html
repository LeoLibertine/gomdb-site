Contexto del Proyecto

Un Query Model transversal
para todos los productos.

El banco está implementando un patrón CQRS donde las operaciones de escritura van al core (iSeries) y las consultas de lectura se sirven desde MongoDB Atlas.

Caso de uso

Patrón CQRS para Core Bancario

Separación de comandos (escrituras al iSeries) y consultas (lecturas desde MongoDB). El Query Model materializa cuentas, saldos y movimientos de todos los productos del banco en un modelo unificado.

Primer producto

Bonos — en 3 meses a producción

Bonos es el primer producto que entra al nuevo modelo dentro de la Plataforma de Desarrollo de Negocio. Seguirán CDT e Inversión Virtual. Luego, Depósitos migra desde el QM de contingencia actual.

Modelo de datos

Cuentas + Postings (Movimientos)

Dos colecciones principales: cuentas con saldos por tipo, y postings con los movimientos de cada cuenta. Modelo canónico reutilizable para todos los productos del core bancario.

Arquitectura CDC

Eventos vía Kafka — misma base del QM actual

Operaciones modifican el core y publican eventos en Kafka (broker empresarial). El microservicio consumidor materializa en MongoDB. Misma arquitectura que el QM de contingencia, orientada a eventos.

Objetivo estratégico: Este QM no es solo para Bonos. Es el chasis reutilizable para todos los productos del banco. Lo que se diseñe hoy debe estar preparado para escalar cuando Depósitos (~20M cuentas, ~25M movimientos/día) migre a este modelo.

Consideraciones Técnicas

Lo que hay que definir
en el working session.

Decisiones de diseño que impactan el rendimiento y la evolución del cluster.

Shard Key

Sharding por accountId (hashed)

Distribución uniforme de cuentas entre shards. Soporta el patrón de acceso principal (consultas por cuenta). Evita hotspots de productos con muchas cuentas (ahorro, corriente). Si se inicia sin sharding, la migración posterior tiene cierta complejidad — evaluar trade-offs.

Indexes

Estrategia de índices por colección

Cuentas: _id + parameters.mdmKey. Postings: accountId + requestId. Considerar compuesto: {accountId: 1, valueTime: -1} para consultas por rango de fecha. Índice en transactionId para tracking.

Online Archive

S3 después de 3-6 meses

Movimientos con más de 3-6 meses (por definir) van a Online Archive (S3). Reduce costo de almacenamiento ~90%. Latencia aceptable para consultas históricas. Campo de archivado: valueTime o insertionTime.

Connection Pool

2,500 conexiones vía mongos

50 pods × 50 conexiones = 2,500. Las conexiones van al mongos router, que las distribuye entre shards. Tuning: maxPoolSize: 50, minPoolSize: 10. Mongos maneja el ruteo transparente a cada shard.

CDC Pipeline

Idempotencia y deduplicación

Usar requestId como clave de deduplicación en operaciones upsert. Monitorear lag del consumer Kafka — alertar si >5 segundos. Dead letter queue para transformaciones fallidas. Rate limiting en el microservicio consumidor.

Balanceo

Distribución de chunks entre shards

Con accountId: "hashed", los chunks se distribuyen uniformemente. Monitorear sh.status() para validar balance. Si un shard acumula más chunks (ej. por inserts masivos de un producto), el balancer de Atlas redistribuye automáticamente.

Configuración de Sharding

Shard key seleccionada
y opciones evaluadas.

El equipo confirmó: "shard key a nivel cuenta". Aquí la configuración recomendada con la alternativa evaluada para cada colección.

Colección: cuentas
Opción A — Recomendada

Hashed por _id

{ _id: "hashed" }

El _id de cuentas ya ES el accountId (B00000000001, S12345678901). Hashed distribuye uniformemente sin importar el prefijo de producto.

Ventajas
  • Distribución perfecta entre shards — sin hotspots por producto
  • Query por _id va directo a un solo shard (targeted)
  • No requiere índice adicional — el shard key usa el _id existente
  • Los prefijos S/D/B no crean chunks desbalanceados como harían con ranged
Trade-offs
  • Queries cross-account por stakeholderIds son scatter-gather
  • No soporta range queries sobre el shard key (hashed no mantiene orden)
Opción B — Alternativa

Ranged por _id

{ _id: 1 }

Shard key ranged aprovecha que el _id tiene prefijo de producto. Cuentas del mismo tipo quedan en el mismo rango de chunks.

Ventajas
  • Queries por rango de cuentas del mismo producto son eficientes
  • Facilita operaciones de archivado por producto
Trade-offs
  • Riesgo de hotspots: Depósitos (D*) tiene 20M cuentas vs Bonos (B*) 100 — distribución muy desigual
  • Si un producto domina, un shard recibe la mayoría del tráfico
  • Requiere monitoreo activo de balanceo de chunks
Colección: postings
Opción A — Recomendada

Hashed por accountId

{ accountId: "hashed" }

Todos los movimientos de una cuenta quedan en el mismo shard. Query principal (movimientos de una cuenta) es targeted.

Ventajas
  • Query {accountId: X} va directo a un shard — colocación de datos
  • Distribución uniforme entre shards
  • Consistente con el shard key de cuentas — misma lógica de ruteo
  • El consumer CDC inserta por cuenta — writes son targeted
Trade-offs
  • Query "todos los movimientos de HOY" (sin filtro de cuenta) es scatter-gather
  • Range scan por valueTime sin filtro de accountId es scatter-gather
Opción B — Alternativa

Compound accountId + valueTime

{ accountId: 1, valueTime: 1 }

Compound key que mantiene colocación por cuenta y además permite range queries por fecha dentro de esa cuenta.

Ventajas
  • Queries {accountId: X, valueTime: {$gte, $lte}} son targeted y eficientes
  • Soporta el patrón "últimos N movimientos de la cuenta"
  • Facilita Online Archive por fecha (archivado por valueTime)
Trade-offs
  • Sin hash — cuentas hot (ahorro/corriente en Fase 2) crean chunks grandes
  • Distribución menos uniforme que hashed
  • La shard key compound es immutable — no se puede cambiar después
  • Más complejo de operar: requiere monitoreo de chunk splits
¿Por qué sharding desde día uno?
RazónDetalle
Evita migración futura Migrar de non-sharded a sharded en producción con datos y tráfico requiere sh.shardCollection() con redistribución de chunks. Tiene impacto en performance durante la migración. Habilitarlo desde el inicio elimina este riesgo.
Costo comparable 2 shards de M40 (6 nodos × 16 GB) tiene un costo similar a 1 replica set de M50 (3 nodos × 32 GB). Misma RAM total (32 GB), pero con escalabilidad horizontal.
Fase 2 es inevitable Depósitos va a migrar al QM. Con 22M cuentas y 30M mov/día, sharding será necesario. Mejor tener la arquitectura lista que migrar bajo presión.
Mejor fault isolation Con 2 primaries independientes, un problema en un shard solo afecta la mitad de las cuentas. En replica set único, el primary es single point of failure para writes.
Picos distribuidos Nóminas y fin de mes distribuyen la carga entre shards. Un solo replica set absorbe todo el pico. El QM es path primario sin fallback — distribuir la carga reduce el riesgo.
Configuración seleccionada: accountId: "hashed" para ambas colecciones (cuentas usa _id que ya es el accountId). Distribución uniforme, queries targeted por cuenta, writes del CDC targeted. El trade-off de scatter-gather en queries cross-account por titular es aceptable — representan una fracción menor del tráfico (B1-Q13: consultas por cuenta individual).
