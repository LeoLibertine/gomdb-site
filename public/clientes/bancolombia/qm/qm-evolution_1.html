Entendimiento Actual

Lo que vemos hoy
en el Query Model.

Basado en nuestras sesiones con el equipo de arquitectura del banco.

Patrón CQRS

Escrituras al iSeries. Lecturas desde MongoDB. ~70% lecturas, ~30% escrituras.

Dominios de Datos

Materializa relación, cuenta, cliente, movimientos, saldos y cuentas escritas para Depósitos.

Rol en Contingencia

Switch entre iSeries y QM. El objetivo: leer desde QM permanentemente, no solo en contingencia.

Limitaciones Conocidas

Bolsillos, bloqueos parciales y retenciones regulatorias no representados en contingencia.

Expansión: Bonos

Segundo cluster "Core Banking" desplegado. QM expandiéndose a nuevos dominios financieros.

Visibilidad Ejecutiva

Evidencia Interna

Lo que demostró durante
febrero 2026.

Prueba de relevancia — no prueba de que la arquitectura objetivo completa ya esté implementada.

Componente se mantuvo disponible

MongoDB continuó sirviendo lecturas a canales digitales mientras el AS/400 estuvo indisponible por un período extendido.

Cargas de lectura sostenidas

Saldos, historial de transacciones e información de cuentas se siguieron sirviendo a App Personas y canales.

Retroalimentación interna positiva

El equipo de arquitectura dio feedback positivo sobre el rol de MongoDB en este tipo de carga operacional.

Alta visibilidad ejecutiva

Tras el incidente, se evaluó la expansión de capacidades de MongoDB a más servicios y dominios.

Encuadre: Esta evidencia valida al Query Model como componente relevante. La propuesta construye sobre esta base para definir un camino de evolución controlada e intencional.

Lo que funciona hoy está claro.
La pregunta es: ¿hacia dónde evoluciona?

Evolución ordenada del Query Model

✓ Productos nuevos (Molde) se trabajarán greenfield con modelo canónico.

Lo que esto significa: Una cosa es lo que se necesita para la próxima caída. Otra es cómo se diseña una capacidad durable. El Query Model necesita servir al corto plazo sin comprometer el largo plazo.

Riesgo Principal

El mayor riesgo no es invertir poco —
es crecer sin estructura.

El Query Model funcionó bien. Por eso la tentación de "meterle más." Pero sin principios claros, se vuelve otro componente táctico inflado.

Si crece sin gobierno

Se agrega dominio tras dominio sin ownership claro. Cada equipo pide "meter más saldos" y el QM se vuelve un catch-all sin límites.

→ Sin modelo canónico
→ Sin límites de dominio
→ Sin ownership por producto
→ Otro componente táctico inflado

Si crece con intención

Cada expansión sigue principios: qué dominio, qué modelo canónico, quién es dueño, cómo se alimenta. QM evoluciona como capacidad.

✓ Dominio definido
✓ Modelo canónico claro
✓ Ownership asignado
✓ Capacidad estratégica reutilizable
"La evolución correcta no es hacerlo más grande.
Es hacerlo más intencional."

Dirección Objetivo

Datos de producto +
capa compartida de acceso.

Producto con modelo canónico claro, eventos bien definidos y una capa de acceso que no se rompa cada vez que cambia el core.

Dominios /
Productos

Cada producto define su modelo canónico

CDTs, depósitos, pagos, tarjetas — cada uno con ownership claro, datos bien definidos, APIs documentadas. Esto es lo que el banco llama Data Mesh.

Capa de
Eventos

CDC + Kafka + Stream Processing

Cada producto publica sus cambios como eventos. La capa de distribución los consume y materializa. Centralización del acceso, no de los datos.

Query Model
Evolucionado

Capa operacional de lectura — siempre activa

Materializa datos canónicos de cada dominio. Alimentada por CDC/eventos. Sirve a todos los canales digitales por defecto. Ejecuta sobre MongoDB Atlas.

No se trata de etiquetas como "data mesh" o "data fabric." Se trata de que cada producto tenga su data limpia, bien definida, y que exista una capa de acceso operacional que no dependa del core para cada lectura.

04

06

¿Qué se resuelve solo para contingencia vs. qué se resuelve para servicio permanente?
