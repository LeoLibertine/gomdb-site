Contexto Estratégico

Los tres pilares de la
estrategia de datos.

El Query Model no es un componente aislado — es uno de los tres pilares fundamentales de la estrategia de datos del banco.

CDP

Customer Data Platform
Pilar analítico. Segmentación, perfilamiento de clientes y modelos predictivos.

Data Access Layer

Pilar Operacional
APIs y servicios de acceso a datos. Capa de abstracción para consumidores.

Query Model

Pilar Transaccional
Consultas en tiempo real. Lectura optimizada de saldos, movimientos y estado de cuenta.

Visión unificada: Estos tres pilares trabajan de forma complementaria. El QM alimenta la capa transaccional en tiempo real, el DAL expone los datos como APIs operacionales, y el CDP habilita la analítica. La estrategia de gobierno de datos los conecta transversalmente.

Espacio en disco — Resuelto

Inicialmente reportado 16% libre (CRITICAL). Validación actual: ~40% libre. Auto-Expand Storage habilitado. Monitoreo de tendencia 7-14 días recomendado.

WiredTiger Cache — Confirmado

Cache de 82.4 GB, 66 GB en uso. Working set dominado por índices. Colección MovementsEndOfDay: 117 GB + 24 GB índices. processed_events: 42 GB en índices (sin uso actual).

Lock Contention — Confirmado

25.7% del tiempo. Workload dominado por escrituras (5.79B updates en 4 días). Cada update mantiene 29 índices incluyendo los no usados. TTL de processed_events con expireAfterSeconds=0.

12 índices sin uso — 35.2 GB

Parcialmente confirmado. Índices creados para casos históricos. El cluster solo recibe escrituras (réplica), por lo que la baja lectura es esperada. Pendiente profiler 7 días.

Colecciones Principales — Peso en el Cluster
Patrón de queries

Últimos movimientos por cuenta. Nuevos desarrollos filtrarán por fecha + cuenta.

Write concern

Majority. Todas las escrituras replicadas desde el sistema origen con write concern.

Plan de remediación

Depurar índices no usados, validar TTL de processed_events, evaluar sharding pre-fase 2.

Acción inmediata: Depurar los 12 índices sin uso (35.2 GB) reducirá lock contention y liberará WiredTiger cache. Esto es prerequisito antes de escalar el tier o evaluar sharding.

Dos Tracks en Paralelo

Contingencia y modernización
avanzan simultáneamente.

El banco ejecuta dos líneas de trabajo que convergen en el futuro en una sola arquitectura QM.

Track 1

QM Contingencia → Acceso Permanente

"Evolucionar lo que ya funciona"

Estado actual: QM de contingencia ante caídas del iSeries

Evolución: de contingencia a capa de acceso permanente en tiempo real

Pipeline: iSeries → CDC Extractor → Kafka → Stream Processor → Atlas

Alcance: ~20M cuentas de depósitos + productos existentes

Pendiente: post-review del QM actual (colecciones, modelo, índices)

⚠ Reducir dependencia directa del core para consultas de lectura.
Track 2

QM Core Modernizado

"Lo que hacemos en Bonos sienta las bases para todo"

Nuevo core bancario arranca con producto Bonos

Pipeline: Nuevo Core → CDC → Kafka → Stream Processor → Atlas

Colecciones: Account State (cuentas) + Postings (movimientos)

Sizing inicial: ~100K cuentas, crecimiento a millones

Greenfield: modelo canónico desde el inicio

✓ Bonos como piloto sienta las bases para CDT, Inversiones y más.
Convergencia

Ambos tracks convergen en una
arquitectura QM unificada.

Cuando Depósitos migre al nuevo core, el QM de contingencia se retira. El QM modernizado absorbe todo.

Roadmap de Evolución — Convergencia de Tracks
Fase 1 — Bonos

Piloto en nuevo core con modelo canónico. ~100K cuentas iniciales, Account State + Postings.

Fase 2 — Expansión

CDT e Inversión Virtual migran. QM Contingencia reduce su alcance progresivamente.

Fase 3 — Depósitos

Cuando Depósitos migra al nuevo core, el QM Contingencia se retira completamente.

Final — Unificado

Una sola arquitectura QM para todos los productos. Gobierno por dominio. Data Products.

Capacidades Habilitadas

Lo que el QM evolucionado
habilita para el banco.

Más allá de contingencia — un QM maduro desbloquea capacidades estratégicas para toda la organización.

Consulta en Tiempo Real

Saldos, movimientos y estado de cuenta sin impactar el core. Lectura permanente desde MongoDB Atlas.

Autorización Mejorada

Reducción del tiempo de respuesta en autorizaciones cross-producto. Datos pre-materializados listos para consulta.

Resiliencia como Diseño

QM como path primario de lectura — no como fallback. Disponibilidad 99.9% sin depender del core para cada lectura.

Snapshots de Saldos

Nueva colección de snapshots diarios para conciliación. ~20-30M cuentas × 60 días de retención hot.

Archivo Inteligente

Online Archive a S3 después de 3-6 meses hot. Datos históricos accesibles sin impactar performance del cluster.

Gobierno por Dominio

Cada dominio bancario (Bonos, CDT, Depósitos) responsable de sus propios datos. Alineado a Data Mesh / Data Products.

Latencia CDC objetivo: ≤200ms. SLA objetivo: 99.9%. Sharding: No necesario fase 1, planificado antes de fase 2 (accountId hashed). Operación: Equipo interno del banco.
Riesgo Principal

El mayor riesgo no es invertir poco —
es crecer sin estructura.

El Query Model funcionó bien. Por eso la tentación de "meterle más." Pero sin principios claros, se vuelve otro componente táctico inflado.

Si crece sin gobierno

Se agrega dominio tras dominio sin ownership claro. Cada equipo pide "meter más saldos" y el QM se vuelve un catch-all sin límites.

→ Sin modelo canónico
→ Sin límites de dominio
→ Sin ownership por producto
→ Otro componente táctico inflado

Si crece con intención

Cada expansión sigue principios: qué dominio, qué modelo canónico, quién es dueño, cómo se alimenta. QM evoluciona como capacidad.

✓ Dominio definido
✓ Modelo canónico claro
✓ Ownership asignado
✓ Capacidad estratégica reutilizable
"La evolución correcta no es hacerlo más grande.
Es hacerlo más intencional."
Por Qué Ahora

La presión de expandir ya existe.
La pregunta es si crece ordenado.

El éxito del Query Model en febrero generó tracción interna. Hay misiones activas para habilitar más capacidades. Si se expande sin frontera clara, se vuelve otro componente táctico inflado.

Misiones Activas

Presión de Crecimiento

Se quiere meter más saldos y más dominios al QM — la demanda de expansión ya es real.

Riesgo de Desorden

Sin gobierno, cada equipo agrega a su manera. El QM se vuelve un catch-all sin límites.

Ventana de Decisión

Producto con modelo canónico claro, eventos bien definidos y una capa de acceso que no se rompa cada vez que cambia el core.

Dominios /
Productos

Cada producto define su modelo canónico

Depósitos, bonos, pagos, tarjetas — cada uno con ownership claro, datos bien definidos, APIs documentadas. Alineado con la visión de data mesh y data fabric que el banco está evaluando.

Capa de
Eventos

CDC + Kafka + Stream Processing

Cada producto publica sus cambios como eventos. La capa de distribución los consume y materializa. Centralización del acceso, no de los datos.

Query Model
Evolucionado

Capa operacional de lectura — siempre activa

Materializa datos canónicos de cada dominio. Alimentada por CDC/eventos. Sirve a todos los canales digitales por defecto. Ejecuta sobre MongoDB Atlas.

No se trata de etiquetas como "data mesh" o "data fabric." Se trata de que cada producto tenga su data limpia, bien definida, y que exista una capa de acceso operacional que no dependa del core para cada lectura.
Ejemplo real: Bonos necesita datos de cuenta (capa de sistema) + datos de instrumento financiero (dominio inversiones). Hoy no tienen dónde juntarlos limpiamente. Un QM evolucionado resuelve esto como capa compartida de acceso.
