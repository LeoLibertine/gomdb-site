SPs Lambda Carga Batch

Análisis de los stored procedures utilizados para la carga masiva y su migración al Microservicio de Carga Batch en AWS Lambda.

📦 Contexto de Carga Batch

Los stored procedures de carga batch están diseñados para el procesamiento masivo de datos desde archivos programados. Su lógica se migra al Microservicio de Carga Batch que se ejecuta mediante AWS Lambda de forma programada.

📦

dbo.spInsertarDataLiquida

Motor para carga batch

Este SP es muy similar a spInsertarDataLiquidaEvo, pero está optimizado para carga batch controlada donde los datos ya vienen limpios y validados. No tiene las validaciones iniciales de la versión "Evo".

1

Lógica de Mantenimiento Físico

Su primera acción es llamar a EXEC [API].[spMantenerGrupos]. Esto confirma que su lógica está fuertemente acoplada a la gestión de particiones físicas de SQL Server.

Líneas de Código: EXEC [API].[spMantenerGrupos]
2

Preparación y Cálculo de Datos

Realiza la misma lógica de negocio para calcular FechaVigencia, NumDiasVigencia y FechaDepuracion, incluyendo el LEFT JOIN a la tabla DiasVigenciaTipoDocumento.

Líneas de Código: LEFT JOIN DiasVigenciaTipoDocumento, cálculo de fechas
3

Efecto Secundario (Tracking)

Ejecuta una sentencia UPDATE en la tabla [UltimaExtraccionProcesoDocumento]. Es una tabla de control para que los procesos de ETL sepan cuál fue la última fecha de ingesta.

Líneas de Código: UPDATE [UltimaExtraccionProcesoDocumento]
4

Inserción del Registro

Inserta el registro final en la tabla DatosLiquidos después de realizar todos los cálculos, sin validaciones previas.

Líneas de Código: INSERT INTO DatosLiquidos

🔍 Diferencias Clave con spInsertarDataLiquidaEvo

La ausencia de bloques de validación (IF NOT EXISTS...) sugiere que este SP es utilizado por un proceso controlado (como la Carga Batch), donde se asume que los datos ya vienen limpios y validados.

❌

No Tiene: Validaciones

Sin validación de datos maestros (Entidad, Canal, Tecnología)

➕

Sí Tiene: Tracking

Actualización de tabla de control de última extracción

🎯

Propósito: Batch

Diseñado para procesos controlados de carga masiva

🔄 Transformación a la Arquitectura de MongoDB

La estrategia de migración es idéntica a la del SP ...Evo, pero aplicada en el contexto del Microservicio de Carga Batch.

Lógica en el Stored Procedure (SQL Server)
Cómo se Maneja en el Microservicio de Carga Batch (MongoDB)
1. Llamada a spMantenerGrupos
Eliminado. La gestión de particiones es irrelevante en MongoDB.
2. Cálculo de FechaVigencia y FechaDepuracion
Se mueve al código del Microservicio de Carga Batch. La Lambda leerá los datos del archivo de entrada, aplicará las mismas funciones de cálculo para las fechas y construirá el documento JSON final.
3. UPDATE a UltimaExtraccionProcesoDocumento
Se reemplaza por una actualización a una colección de control. El microservicio, después de insertar un lote de documentos en registros_bdtl, puede realizar una operación updateOne (con upsert: true) en una pequeña colección de MongoDB llamada tracking_extraccion.
4. Inserción de Datos
Reemplazado por insertMany(). En lugar de insertar fila por fila, el microservicio de carga batch puede construir un arreglo de documentos y usar el comando insertMany() de MongoDB para insertar todo el lote en una única y eficiente operación de red.

⚡ Ejemplo del Flujo en el Microservicio de Carga Batch

1

Activación y Lectura

La Lambda se activa y lee un archivo de S3 con 1000 registros.

2

Iteración en Memoria

Itera sobre cada registro en memoria.

3

Aplicación de Lógica

Para cada registro, aplica la lógica de negocio (cálculo de fechas, etc.) y construye el documento JSON final con la estructura recomendada.

4

Construcción del Arreglo

Añade cada documento a un arreglo.

5

Inserción Masiva

Al final, ejecuta db.collection('registros_bdtl').insertMany(arregloDeDocumentos).

6

Actualización de Tracking

Si la inserción es exitosa, ejecuta db.collection('tracking_extraccion').updateOne(...) para actualizar la fecha del último procesamiento.

dbo.spMantenerGrupos LEGACY

Gestión de particiones SQL Server

❌ Este SP se Elimina por Completo

Este SP es una tarea de administración física exclusiva de SQL Server. Su única función es gestionar el particionamiento de tablas, dividiendo una tabla grande en pedazos más pequeños y distribuyéndolos en diferentes archivos (filegroups).

🚫 ¿Por Qué se Elimina?

Los problemas que este SP intenta resolver (rendimiento, escalabilidad, mantenimiento) se manejan de forma nativa y más eficiente en MongoDB a través de sus propias características.

🎯 Conclusión: spMantenerGrupos es un artefacto 100% legado de la arquitectura SQL Server.

🍃 ¿Cómo se Maneja Esto en MongoDB?

Los problemas que el SP intentaba resolver se manejan de forma nativa, automática y más eficiente en MongoDB:

🔀

Sharding

Para Escalabilidad: MongoDB distribuye automáticamente los datos. Solo defines una "Shard Key" y MongoDB se encarga del resto.

🔍

Índices

Para Performance: La velocidad depende de índices correctos, no de archivos físicos. Con el modelo desnormalizado las consultas son extremadamente rápidas.

⏰

TTL Indexes

Para Depuración: Índices TTL (Time-To-Live) en FechaDepuracion para borrado automático. Reemplaza el Job Depuración y la lógica de particionamiento.

🚀 Ventajas del Microservicio de Carga Batch

La migración de la lógica del SP a un microservicio aporta beneficios significativos para el procesamiento masivo de datos.

⚡

Operación insertMany()

Una sola operación de red para insertar múltiples documentos vs múltiples INSERTs individuales

🔄

Procesamiento Eficiente

Lógica de negocio aplicada en memoria antes de la inserción masiva

📊

Tracking Simplificado

Colección de control simple en MongoDB vs tabla compleja en SQL Server

🛠️

Sin Mantenimiento Físico

Eliminación completa de la gestión manual de particiones y filegroups

🎯 Resumen de la Migración

Al igual que su contraparte "Evo", este SP se desmantela. Su lógica de negocio se traslada al Microservicio de Carga Batch, y sus funciones de mantenimiento de base de datos se reemplazan por las operaciones nativas y más eficientes de MongoDB.

🔄 spInsertarDataLiquida

Se convierte en Microservicio de Carga Batch con lógica optimizada para procesamiento masivo mediante insertMany().

🚫 spMantenerGrupos

Se elimina por completo. Su funcionalidad es reemplazada por características nativas de MongoDB.

✅ Beneficio Neto

MongoDB + Microservicio = procesamiento más eficiente, escalable y mantenible para cargas batch.
