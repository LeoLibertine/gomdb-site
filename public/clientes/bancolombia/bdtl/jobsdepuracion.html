Jobs Depuración

Análisis de los jobs de depuración automática y su transformación a Índices TTL nativos en MongoDB para el mantenimiento automatizado de datos.

🗑️ Contexto de Depuración

Los jobs de depuración actuales son procesos de mantenimiento complejos diseñados para borrar registros antiguos de la BDTL. Su complejidad está directamente ligada a las limitaciones de la arquitectura física de SQL Server.

🗑️

[PS].[spDepuracionBDTL]

Job de depuración automática

Este SP es el "Job de Depuración" principal. Su complejidad está directamente ligada a las limitaciones de la arquitectura física de SQL Server. Su lógica se divide en tres partes principales:

1

Identificación y Borrado en Lotes

Identifica todos los registros cuya FechaDepuracion es anterior o igual a la fecha actual. Luego, entra en un bucle WHILE para borrar los registros en lotes de 20,000 filas para evitar bloqueos de tabla.

Técnica: Bucle WHILE con DELETE TOP(20000) para manejo de eliminaciones masivas
2

Borrado en Tablas Relacionadas

Antes de borrar de DatosLiquidos, también borra los registros correspondientes de la tabla EXPORT_DATA, que era parte de la integración legada con Kofax.

Dependencias: Limpieza en EXPORT_DATA antes del borrado principal
3

Mantenimiento de Particiones Físicas

La parte más compleja y específica de SQL Server. Ejecuta un bucle para ejecutar ALTER PARTITION FUNCTION ... MERGE RANGE, eliminando las particiones físicas del disco que ahora están vacías.

SQL Server: Mantenimiento manual de particiones físicas vacías
📅

[PS].[spDepuracionBDTLbyDate]

Depuración manual por fecha

Este SP es casi una copia exacta del anterior. La única diferencia es que en lugar de usar la fecha actual (GETDATE()), acepta un parámetro de fecha (@FechaDepuracion). Es una versión del job que puede ser ejecutada manualmente para borrar datos hasta una fecha específica.

🎯 Propósito

Permite la depuración manual hasta una fecha específica, útil para tareas de mantenimiento excepcionales o limpieza dirigida de datos.

🔄 Transformación a la Arquitectura de MongoDB

Toda la lógica de estos Stored Procedures se reemplaza por una única y simple característica nativa de MongoDB: los Índices TTL (Time-To-Live).

Lógica en el Stored Procedure (SQL Server)
Cómo se Maneja en el Microservicio (MongoDB)
1. Borrado en Lotes (WHILE loop)
Reemplazado por el proceso de fondo de TTL. MongoDB tiene un proceso automático en segundo plano que busca y elimina documentos expirados de manera eficiente sin necesidad de lotes manuales.
2. Mantenimiento de Particiones (MERGE RANGE)
Eliminado. Este concepto no existe en MongoDB. La gestión del espacio en disco es automática.
3. Job Externo (SQL Agent)
Eliminado. No se necesita un job externo para ejecutar el borrado. El índice TTL es una característica "siempre activa" de la base de datos.

⏰ La Solución: Índices TTL (Time-To-Live)

Un índice TTL es una característica especial de MongoDB diseñada para este caso de uso exacto: borrar automáticamente documentos después de un cierto tiempo.

🔧 ¿Cómo funciona?

1. Se crea un índice especial en un campo de fecha
2. Se configura la expiración en el campo FechaDepuracion
3. MongoDB borra automáticamente documentos expirados

💻 ¿Cómo se implementaría?

Se ejecutaría un único comando en la colección registros_bdtl, una sola vez:

📝 Implementación del Índice TTL

// Crear el índice TTL en el campo FechaDepuracion db.collection('registros_bdtl').createIndex( { "FechaDepuracion": 1 }, // El campo de fecha { expireAfterSeconds: 0 } // Expira 0 segundos después de la fecha indicada en el campo );

🎯 ¡Y eso es todo! MongoDB se encargará de borrar los documentos de forma automática y eficiente una vez que la fecha en el campo FechaDepuracion haya pasado.

📅 Para la Depuración Manual (Caso Excepcional)

Si en alguna situación excepcional un administrador necesitara borrar manualmente los registros hasta una fecha específica, no se necesitaría un SP complejo.

🔧 Nuevo Enfoque: Microservicio de Administración

Un Microservicio de Administración (AWS Lambda) expondría un endpoint seguro que, al ser llamado con una fecha, ejecutaría un único y simple comando en MongoDB:

// Comando para borrar manualmente todos los documentos hasta una fecha específica const fechaManual = new Date("2025-01-15T00:00:00Z"); await db.collection('registros_bdtl').deleteMany({ "FechaDepuracion": { "$lte": fechaManual } });

⚖️ SQL Server vs MongoDB: Gestión de Depuración

❌ SQL Server (Actual)

  • SP complejo de +100 líneas
  • Bucles WHILE para borrado en lotes
  • Mantenimiento manual de particiones
  • Job externo programado (SQL Agent)
  • Borrado en tablas relacionadas
  • Gestión de bloqueos de tabla
VS

✅ MongoDB (Nuevo)

  • Un único comando de configuración
  • Proceso automático en segundo plano
  • Sin mantenimiento de particiones
  • Característica "siempre activa"
  • Sin dependencias de tablas
  • Sin riesgo de bloqueos

🎯 Conclusión sobre Jobs de Depuración

Un Stored Procedure complejo de más de 100 líneas, que requiere un job externo para ejecutarse y realiza un mantenimiento físico manual, se reemplaza por un único comando de configuración en MongoDB.

🌟 Resultado Final

A partir de ese momento, la depuración de datos se vuelve completamente automática, más eficiente y no requiere ninguna lógica en la capa de aplicación.

🔄 Depuración Automática (99%)

Índice TTL maneja el borrado diario de forma automática y eficiente

📅 Depuración Manual (1%)

Microservicio de Administración con simple deleteMany() para casos excepcionales

✨ Beneficio Neto

MongoDB resuelve la depuración de manera nativa, automática y más eficiente
