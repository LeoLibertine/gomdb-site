From BDTL - Store Procedures

Análisis de los stored procedures de consulta y administración utilizados por el Front BDTL y su migración a microservicios de administración con MongoDB.

🔍

[API].[spConsultarTParametrosEditable] SIMPLE

Consulta de parámetros editables

Este SP tiene una única y simple función:

🎯 Qué hace

Ejecuta una consulta SELECT sobre la tabla de configuración PS.TParametros y devuelve únicamente aquellos parámetros que están marcados como Editable = 'True'.

🖥️ Propósito

Este SP es utilizado por el componente "Front BDTL". Su objetivo es alimentar una interfaz de administración donde un usuario puede ver y modificar ciertos parámetros de configuración del sistema.

🔄 Transformación a la Arquitectura de MongoDB y Microservicios

La lógica de este SP se migra directamente a la capa de aplicación, específicamente al Microservicio de Administración.

Lógica en el Stored Procedure (SQL Server)
Cómo se Maneja en el Microservicio (MongoDB)
1. SELECT * FROM PS.TParametros
Reemplazado por db.collection.find(). La tabla PS.TParametros se convierte en una colección en MongoDB, que podríamos llamar parametros_configuracion.
2. WHERE Editable = 'True'
Reemplazado por un Objeto de Filtro. La cláusula WHERE se convierte en un simple objeto de filtro en la consulta find().

💻 Ejemplo: La Nueva Consulta en MongoDB

El "Front BDTL", en lugar de llamar al SP, ahora llamaría a un endpoint del Microservicio de Administración. Ese microservicio ejecutaría la siguiente consulta en MongoDB:

// La consulta que reemplaza al Stored Procedure const resultados = await db.collection('parametros_configuracion').find({ "Editable": true // El filtro es directo y simple }).toArray();

📋 En resumen

Este es un caso de migración muy directo. El Stored Procedure, que actuaba como una simple capa de acceso a datos, se reemplaza por una consulta find() nativa de MongoDB dentro del microservicio correspondiente. Esto sigue el principio de mover toda la lógica, incluso las consultas simples, a la capa de aplicación para mantener la base de datos como un sistema de almacenamiento puro.

🔍

API.spConsultarTParametrosEditableId SIMPLE

Consulta por ID específico

🎯 Qué hace

Ejecuta una consulta SELECT sobre la tabla PS.TParametros para encontrar un único registro que coincida con un @IdParametro específico y que además esté marcado como Editable = 'True'.

🖥️ Propósito

Es utilizado por el "Front BDTL". Cuando un administrador hace clic en un parámetro específico de la lista (que fue obtenida con el SP anterior) para ver sus detalles o editarlo, esta es la consulta que se ejecuta para traer la información de ese parámetro en particular.

🔄 Transformación a la Arquitectura de MongoDB y Microservicios

La migración es igual de directa que la del SP anterior. La lógica se traslada al Microservicio de Administración.

Lógica en el Stored Procedure (SQL Server)
Cómo se Maneja en el Microservicio (MongoDB)
1. SELECT * FROM PS.TParametros
Reemplazado por db.collection.findOne(). Se utiliza findOne() porque sabemos que estamos buscando un único documento.
2. WHERE Editable = 'True' AND IdParametro=@IdParametro
Reemplazado por un Objeto de Filtro. La cláusula WHERE se convierte en un objeto de filtro con dos condiciones.

💻 Ejemplo: La Nueva Consulta en MongoDB

El "Front BDTL" llamaría a un endpoint del Microservicio de Administración, pasándole el ID del parámetro (ej. /parametros/25). El microservicio ejecutaría la siguiente consulta:

// El ID del parámetro que llega desde la URL const idParametro = 25; // La consulta que reemplaza al Stored Procedure const resultado = await db.collection('parametros_configuracion').findOne({ "IdParametro": idParametro, "Editable": true });

📋 En resumen

De nuevo, es una migración directa. El SP se reemplaza por una consulta findOne() muy simple y eficiente dentro del microservicio, siguiendo el principio de centralizar toda la lógica de acceso a datos en la capa de aplicación.

📝

API.spConsultarTParametrosEditableName SIMPLE

Consulta por nombre de parámetro

🎯 Qué hace

Ejecuta una consulta SELECT sobre la tabla PS.TParametros para encontrar un registro que coincida con un nombre específico (@NombreParametro en el campo ConfigurationFilter) y que también esté marcado como Editable = 'True'.

🖥️ Propósito

Es utilizado por aplicaciones o servicios que necesitan obtener un valor de configuración conociendo su nombre en lugar de su ID. Por ejemplo, un servicio podría necesitar obtener la URL del "ServidorLDAPAPI" para conectarse.

🔄 Transformación a la Arquitectura de MongoDB y Microservicios

La migración sigue el mismo patrón que los SPs anteriores. La lógica se traslada al Microservicio de Administración o a cualquier otro microservicio que necesite leer la configuración.

Lógica en el Stored Procedure (SQL Server)
Cómo se Maneja en el Microservicio (MongoDB)
1. SELECT * FROM PS.TParametros
Reemplazado por db.collection.findOne().
2. WHERE Editable = 'True' AND [ConfigurationFilter]=@NombreParametro
Reemplazado por un Objeto de Filtro.

💻 Ejemplo: La Nueva Consulta en MongoDB

Un microservicio que necesite este parámetro llamaría a una función que ejecutaría la siguiente consulta:

// El nombre del parámetro que se necesita const nombreParametro = "ServidorLDAPAPI"; // La consulta que reemplaza al Stored Procedure const resultado = await db.collection('parametros_configuracion').findOne({ "ConfigurationFilter": nombreParametro, "Editable": true });

📋 En resumen

Es una migración directa. El SP se reemplaza por una consulta findOne() simple y eficiente dentro del microservicio, centralizando la lógica de acceso a datos en la capa de aplicación.

API.spInsertarParametrizacionVigencia

Actualización completa de parámetros

🎯 Qué hace

Este SP realiza una actualización completa (borrado e inserción) de la tabla de parámetros [dbo].[DiasVigenciaTipoDocumento]. Esta tabla es la que define cuántos días es válido un tipo de documento para un proceso específico, y es la que se consulta durante la ingesta para calcular el campo FechaVigencia.

🔧 Lógica Clave

1. Inicia una transacción para asegurar que la operación sea atómica.
2. Borra por completo todos los registros existentes en la tabla con TRUNCATE TABLE.
3. Inserta en lote todos los nuevos registros que llegan en el parámetro de tabla @ParametrosVigencia.

🖥️ Propósito

Es utilizado por el "Front BDTL" para que un administrador pueda cargar o actualizar todas las reglas de vigencia de los documentos de una sola vez.

🔄 Transformación a la Arquitectura de MongoDB y Microservicios

La lógica de este SP se migra al Microservicio de Administración, que será el encargado de gestionar todas las colecciones de parámetros.

Lógica en el Stored Procedure (SQL Server)
Cómo se Maneja en el Microservicio (MongoDB)
1. TRUNCATE TABLE
Reemplazado por db.collection.deleteMany({}). Esta operación borra todos los documentos de la colección de parámetros parametros_vigencia.
2. INSERT desde un TVP
Reemplazado por db.collection.insertMany(). El microservicio recibirá un arreglo de objetos JSON en el cuerpo de la petición y los insertará todos en una única y eficiente operación.
3. BEGIN TRAN...COMMIT TRAN
Reemplazado por una Sesión de Transacción de MongoDB. Para garantizar la misma atomicidad (que la operación completa falle o tenga éxito), las operaciones deleteMany e insertMany se ejecutan dentro de una sesión de transacción.

💻 Ejemplo: La Nueva Lógica en el Microservicio de Administración

El "Front BDTL" llamaría a un endpoint (ej. POST /parametros/vigencia) del microservicio, enviando el nuevo conjunto de reglas. El microservicio ejecutaría una lógica similar a esta:

// El arreglo de nuevos parámetros que llega en el cuerpo de la petición const nuevosParametros = [ { "Proceso": "Embargos", "TipoDocumento": "Embargos Genericos", "DiasVigencia": 18250, "UsuarioActualizacion": "user_admin" }, { "Proceso": "Embargos", "TipoDocumento": "Embargos Masivos", "DiasVigencia": 18250, "UsuarioActualizacion": "user_admin" } // ... más parámetros ]; // Iniciar una sesión para la transacción const session = client.startSession(); try { // Iniciar la transacción await session.withTransaction(async () => { const collection = db.collection('parametros_vigencia'); // 1. Borrar todos los documentos existentes dentro de la transacción await collection.deleteMany({}, { session }); // 2. Insertar todos los nuevos documentos dentro de la transacción await collection.insertMany(nuevosParametros, { session }); }); console.log("La parametrización de vigencia se actualizó correctamente."); } catch (error) { console.error("Error en la transacción:", error); // La transacción se aborta automáticamente si hay un error } finally { // Finalizar la sesión await session.endSession(); }

📋 En resumen

La lógica de negocio se traslada a la capa de aplicación. Las operaciones de borrado e inserción masiva se reemplazan por los comandos nativos y eficientes de MongoDB (deleteMany, insertMany), y la atomicidad de la operación se garantiza utilizando las transacciones de MongoDB.
