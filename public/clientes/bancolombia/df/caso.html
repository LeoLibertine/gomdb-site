El Desafío de Bancolombia

Un nuevo sistema que reemplaza la solución actual con capacidades superiores

🎯 Objetivo Principal

Crear un sistema de monitoreo transaccional robusto que procese todas las transacciones de canales digitales, las enriquezca con datos adicionales, y las evalúe en tiempo real contra motores de detección de fraude.

⚡ Requerimientos Críticos

El sistema debe manejar un volumen masivo de 400-500 transacciones por segundo, almacenar hasta 600 campos por transacción, y soportar consultas de 1,200 asesores trabajando 24/7.

Flujo del Sistema

Recepción de Transacciones

Todas las transacciones de canales digitales llegan al sistema en tiempo real

Enriquecimiento de Datos

Se transforman, normalizan y agregan datos adicionales a cada transacción

Análisis de Riesgo

Las transacciones enriquecidas se envían al motor de riesgos para evaluación

Decisión y Almacenamiento

Se recibe la respuesta (aprobar/declinar) y se almacena todo en MongoDB

Consulta por Asesores

Los asesores pueden consultar el historial para gestionar casos de fraude

Estrategia de Sharding

🔑 Shard Key: customer_id

Distribuye los datos de manera equilibrada y co-localiza las transacciones de cada cliente en el mismo shard para consultas eficientes.

📊 Índices Compuestos

Índices optimizados para los patrones de consulta: {customer_id: 1, created_at: -1} y {transaction_id: 1}.

¿Por Qué MongoDB es Perfecto?

Las ventajas clave que hacen de MongoDB la elección ideal

⚡

Alta Performance de Escritura

MongoDB está optimizado para manejar 500+ transacciones por segundo con su arquitectura de escritura en memoria y persistencia asíncrona.

🔧

Esquema Flexible

Evolución de 300 a 600 campos sin migraciones costosas. MongoDB permite agregar nuevos campos sin afectar el rendimiento.

📈

Escalabilidad Horizontal

Sharding nativo permite manejar 10TB+ de datos distribuyendo la carga across múltiples servidores de manera transparente.

🎯

Consultas Optimizadas

Las consultas por customer_id + fecha son extremadamente eficientes gracias al co-location de datos en el mismo shard.

💾

Modelo de Documentos

Cada transacción es una entidad autocontenida, eliminando la necesidad de JOINs complejos entre múltiples tablas.

🛡️

Alta Disponibilidad

Replica Sets garantizan 99.99% uptime, crítico para un sistema de detección de fraude que opera 24/7.

MongoDB vs Alternativas

MongoDB

  • ✅ Esquema flexible
  • ✅ Sharding nativo
  • ✅ Alta escritura
  • ✅ Documentos JSON

SQL Tradicional

  • ❌ Migraciones costosas
  • ❌ JOINs complejos
  • ⚠️ Sharding manual
  • ⚠️ Escalado vertical

Cassandra/HBase

  • ✅ Alta escritura
  • ❌ Consultas limitadas
  • ❌ Complejidad operacional
  • ❌ Curva de aprendizaje
