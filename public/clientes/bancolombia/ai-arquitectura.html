Análisis comparativo de arquitecturas para implementar MongoDB Atlas Vector Search con Voyage AI

📌 Importante: Voyage AI y Amazon Bedrock

Voyage AI no está disponible actualmente en Amazon Bedrock.

Amazon Bedrock ofrece modelos de AI21 Labs, Anthropic, Cohere, Meta, Stability AI y Amazon Titan, pero Voyage AI se consume a través de su propia API independiente.

Implicación: Las llamadas para crear los embeddings se realizarán directamente a la API de Voyage AI, no a través del endpoint de Bedrock. Esto no impide su uso, solo cambia el punto de integración.

Arquitectura 1: Modelo Externo

✓ Inmediatamente Implementable

Descripción General

Tu aplicación (la función Lambda) orquesta el proceso de vectorización antes de almacenar los datos. Es la arquitectura más común y flexible disponible hoy.

🔄 Flujo de Trabajo

1

Generación del Informe

La AWS Lambda genera el informe de rendimiento en formato JSON mediante consultas a las bases de datos.

↓
2

Vectorización Externa

La Lambda hace una llamada API directa a Voyage AI. Envía el contenido del JSON y recibe el vector (embedding) numérico.

↓
3

Almacenamiento en Atlas

La Lambda agrega el vector recibido al documento JSON original y lo escribe en MongoDB Atlas.

↓
4

Consulta del Agente IA

El agente vectoriza su pregunta y usa $vectorSearch en Atlas para encontrar documentos relevantes.

✅ Ventajas

✓
Máxima Simplicidad

Tu código de ingesta es trivial. Solo necesitas guardar el JSON. La complejidad de la vectorización es abstraída por la base de datos.

✓
Latencia Mínima

No hay llamadas de red externas durante la ingesta, resultando en un proceso de guardado mucho más rápido y eficiente.

✓
Seguridad Centralizada

Los datos nunca abandonan el entorno seguro de Atlas para ser vectorizados. Gestión unificada de modelos y datos.

✓
Menor Complejidad Operativa

Menos componentes que monitorear, menos puntos de fallo, menos código que mantener.

❌ Desventajas

✗
Dependencia del Roadmap

Esta funcionalidad está en el roadmap de MongoDB. Aunque es la dirección estratégica, puede que no esté disponible para la PoC inicial.

✗
Menor Flexibilidad (Temporal)

Una vez integrado, cambiar de modelo de embedding requerirá esperar actualizaciones de la plataforma.

⚠️
Disponibilidad Futura

No hay fecha confirmada de disponibilidad general, aunque MongoDB ha confirmado que está en desarrollo activo.

🚀

Fase 1: PoC

Inmediato (2-3 semanas)

Arquitectura 1: Modelo Externo

Por qué: Te permite validar el caso de uso inmediatamente sin esperar funcionalidades futuras.

Objetivos de la PoC:
  • ✓ Validar calidad de embeddings de Voyage AI
  • ✓ Medir rendimiento de Vector Search en Atlas
  • ✓ Comparar vs solución actual (S3 + Bedrock)
  • ✓ Demostrar valor antes de diciembre
🎯

Fase 2: Producción

Objetivo a largo plazo

Arquitectura 2: Modelo Integrado

Por qué: Ofrece rendimiento superior, simplicidad operativa y mayor seguridad.

Beneficios en Producción:
  • ✓ Código más simple y mantenible
  • ✓ Menor latencia de ingesta
  • ✓ Máxima seguridad de datos
  • ✓ Costos operativos optimizados

💡 Estrategia Recomendada

Diseña la PoC con Arquitectura 1 para validación inmediata, pero mantén en mente la transición a Arquitectura 2 como objetivo final. Esta estrategia te permite avanzar sin demoras mientras te preparas para una solución optimizada a largo plazo.

⚙️ Consideraciones Técnicas Clave

📊 Dimensionamiento

Información necesaria de Bancolombia:

  • Volumen de informes JSON por semana/mes
  • Tamaño promedio de cada informe
  • Número de consultas concurrentes esperadas
  • Requisitos de latencia (P95, P99)
  • Retención de datos históricos
  • Crecimiento proyectado

🔒 Seguridad

Aspectos a considerar:

  • Encriptación en tránsito y en reposo
  • Control de acceso basado en roles (RBAC)
  • Auditoría de accesos
  • VPC peering o Private Link
  • Cumplimiento normativo bancario
  • Gestión de secretos (API keys)

📈 Monitoreo

Métricas críticas:

  • Latencia de ingesta (con/sin vectorización)
  • Latencia de consultas vector search
  • Tasa de éxito/error en llamadas API
  • Uso de recursos (CPU, memoria, disco)
  • Costos por operación
  • Calidad de resultados (relevancia)

💰 Costos

Componentes de costo:

  • MongoDB Atlas (compute + storage)
  • API de Voyage AI (por embedding)
  • AWS Lambda (ejecución)
  • Transferencia de datos
  • Claude via Bedrock (por token)
  • Monitoreo y logging

Arquitectura 2: Modelo Integrado

🚀 Visión Estratégica (Roadmap)

Descripción General

Arquitectura simplificada de alto rendimiento donde MongoDB Atlas se encarga de la vectorización de forma nativa. Esta es la visión futura de la plataforma.

🔄 Flujo de Trabajo

1

Ingesta Directa

La AWS Lambda genera el informe JSON y lo escribe directamente en MongoDB Atlas, sin preocuparse por el vector.

↓
2

Vectorización Automática ⚡

MongoDB Atlas detecta el nuevo documento y automáticamente:

  • Usa el modelo Voyage AI embebido en la plataforma
  • Genera el vector automáticamente
  • Puebla el campo correspondiente del documento
  • Todo ocurre dentro del entorno seguro de Atlas
↓
3

Consulta del Agente IA

Proceso idéntico: el agente usa $vectorSearch para encontrar documentos relevantes.
