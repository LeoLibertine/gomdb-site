1. Resumen Ejecutivo

Este documento presenta un análisis técnico detallado y una recomendación de dimensionamiento para el cluster de MongoDB Atlas que servirá al proyecto "Réplica de Salesforce".

Nuestra propuesta está diseñada para garantizar el cumplimiento de los Acuerdos de Nivel de Servicio (SLA) de negocio, soportar los casos de uso críticos y proporcionar una ruta de escalabilidad predecible y costo-eficiente.

Requerimientos Clave

SLA de Rendimiento
500ms
Crecimiento Proyectado
100 → 400 GB

Casos de Uso Críticos

  • Proceso de Tasación: Requiere acceso instantáneo a documentos en la colección assets
  • Liquidación a Comercios en Tiempo Real: Necesita información de establishments y tasas

Patrón de Carga de Trabajo

La operativa se basa en consultas puntuales de alta frecuencia (findOne por sf_id) y actualizaciones atómicas.

1. Resumen Ejecutivo (continuación)

Recomendación Estratégica por Fases

Fase 1: Lanzamiento (100 GB)

Iniciar con un cluster M40 (16 GB RAM).

Es el requisito mínimo para asegurar el SLA de 500ms bajo la carga de trabajo actual.

Fase 2: Crecimiento a 1 año (400 GB)

Planificar el escalado a un cluster M50 (32 GB RAM).

Para absorber el crecimiento de datos y mantener el rendimiento óptimo.

2. Análisis del Workload

Un dimensionamiento correcto no solo se basa en el volumen de datos, sino en cómo se accede a ellos.

El análisis de su workload revela un patrón que MongoDB maneja con extrema eficiencia, siempre que la infraestructura sea la adecuada.

Consultas Críticas

Caso de Uso 1: Proceso de Tasación

Colección: assets

Consulta: findOne({ "sf_id": "..." })

Requiere acceso rápido a documentos específicos para evaluación de activos.

Caso de Uso 2: Liquidación a Comercios

Colección: accounts

Consulta: findOne({ "sf_id": "..." })

Necesita información de establecimientos y tasas para procesamiento en tiempo real.

2. Análisis del Workload (continuación)

Patrón de Consultas

El patrón dominante es el de búsquedas puntuales (point lookups), que son las más eficientes en MongoDB:

  • Lecturas: findOne por el campo sf_id, correctamente indexado en las colecciones accounts y assets
  • Escrituras: update dirigidos por sf_id y addToSet para modificar arrays embebidos, como la relación entre assets y establishments

Conclusión Clave

El rendimiento de toda la aplicación depende de la velocidad con la que se puedan ejecutar estas búsquedas puntuales.

3. Justificación Técnica: RAM y SLA

Para que una consulta como findOne({ "sf_id": "..." }) se ejecute consistentemente por debajo de 500ms, es un requisito técnico indispensable que el índice utilizado ({ sf_id: 1 }) resida en la memoria RAM del cluster.

Operación en RAM
Instantánea

Tiempo de respuesta mínimo, SLA garantizado

Operación en Disco
Lenta

Latencia significativa, SLA en riesgo

Principio Fundamental

Para cumplir con el SLA de 500ms de forma consistente, la RAM del cluster debe ser igual o mayor al Working Set calculado.

3. Cálculo del Working Set

El Working Set es la cantidad de datos e índices que deben residir en RAM para garantizar el rendimiento óptimo.

Fase Datos Índices (10%) Hot Data (10%) Working Set
Fase 1 100 GB ~10 GB ~10 GB ~20 GB
Fase 2 400 GB ~40 GB ~40 GB ~80 GB

4. Dimensionamiento Fase 1

Cluster de Lanzamiento (Working Set ~20 GB)

Tier RAM vs. Working Set Análisis
M30 8 GB ❌ Insuficiente Con 8 GB, ni los índices (~10 GB) caben en memoria. Alto riesgo de incumplir SLA.
M40 RECOMENDADO 16 GB ✓ Óptimo Los 16 GB permiten mantener todos los índices en memoria y espacio para datos calientes. Garantiza el SLA.

4. Dimensionamiento (continuación)

Conclusión Fase 1

Iniciar con un M40 no es un sobrecoste, es una inversión estratégica para asegurar el éxito y la adopción de la aplicación desde el primer día.

Fase 2: Escalabilidad a 1 Año (Working Set ~80 GB)

Tier RAM vs. Working Set Análisis
M50 RECOMENDADO 32 GB ✓ Parcial Podrá alojar porción significativa de índices y datos calientes. Balance costo-eficiente.

4. Dimensionamiento (continuación)

Nota de Escalabilidad

Atlas permite monitorear el "Cache Usage Ratio".

Si esta métrica indica que una porción insuficiente de las lecturas se sirve desde la RAM, se puede escalar a un M60 (64 GB RAM) con cero tiempo de inactividad.
