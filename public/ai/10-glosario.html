Agente 🚀 agentes
Un LLM que decide y actúa usando herramientas — no solo responde.
+
En una frase
Un agente es un LLM con un loop razonar → actuar → observar. Tiene memoria, planifica, y usa herramientas (APIs, bases de datos) hasta lograr un objetivo.
Ejemplo concreto
"Agéndame una llamada con Bancolombia, cierra el deal" → el agente busca el calendario, encuentra hueco, manda el correo, registra en CRM. 3 herramientas en una orden.
"Un chatbot responde. Un agente decide y ejecuta. Esa es la diferencia y la próxima ola — y para vivir necesita memoria persistente: ahí entra MongoDB."
MCP ReAct Tool use
Alucinación 🤖 LLM
Cuando el LLM inventa información con tono seguro. Lo cura RAG.
+
En una frase
El LLM genera texto plausible aunque no tenga datos reales. No miente — completa con su mejor predicción. Para tu cliente: es inaceptable en banca, salud, legal.
Cómo se cura
Con RAG: le pasas al LLM la información real del cliente antes de que responda. Cero invención, todo trazable a la fuente.
"Por eso ningún cliente serio pone un LLM solo en producción. Lo conecta a sus datos vía RAG — y ese 'sus datos' es donde MongoDB cierra."

Atlas Search 🔎 búsqueda
Búsqueda full-text (palabras exactas) sobre tus datos de Mongo. Basado en Lucene.
+
En una frase
Como Elasticsearch — pero integrado a tu base operacional. Una sola plataforma, un solo cluster. Sin pipeline de sincronización.
Cuándo se usa
Cuando el usuario busca palabras exactas: códigos, nombres, SKUs, frases. Encuentra coincidencias literales.
"Si su cliente busca por keywords exactos, Atlas Search. Si busca por intención, Atlas Vector Search. Si quieren ambos: Hybrid Search."
Vector Search Hybrid Search $rankFusion
Atlas Vector Search 🍃 MongoDB AI
El producto estrella: búsqueda por significado integrada a tus datos de Mongo.
+
En una frase
Compara vectores (embeddings) para encontrar contenido semánticamente similar. Sub-100ms sobre millones de vectores. Vive en el mismo cluster que tu data operacional.
Por qué importa
Es la pieza central de RAG. Sin esto, tu cliente necesita Pinecone aparte + sincronizar con Mongo. Con esto: una sola base, un solo query, sin pipeline ETL.
"Esta es la columna que NADIE más tiene en su categoría. Pinecone solo guarda vectores. Postgres no es operacional a escala. Esto es 1 base = vectores + transacciones + búsqueda."
Vector Embedding HNSW
Automated Embedding 🍃 MongoDB AI
MongoDB genera los embeddings automáticamente al insertar datos. Sin pipeline externo.
+
En una frase
Insertas un documento, MongoDB usa Voyage AI por debajo para crear el embedding y guardarlo. Hasta en queries: pasas texto crudo, no vector.
Status
Public Preview en Atlas y Community Edition (anuncio de mayo 2026). Modelos disponibles: voyage-4-large, voyage-4, voyage-4-lite, voyage-code-3.
"Es como auto-pilot para embeddings. Sin pipeline ETL, sin manejo de rate limits, sin context window headaches. Antes era una pesadilla operativa — ahora es un campo más en tu insert."
Voyage AI Embedding
Chunking 📚 RAG
Partir un documento en pedazos manejables antes de vectorizar.
+
En una frase
Cortas el documento en trozos (~500 tokens típico, con 10-20% de solape) porque el LLM no puede leer 200 páginas de un jalón, y vectorizar todo junto pierde precisión.
El secreto
Mal chunking = mal RAG. Cortar a la mitad de una oración pierde contexto. Semantic chunking (cortar por significado) sube precisión hasta 70%.
"El chunking es donde el 80% de los RAG fallan en producción. Es invisible para el cliente pero define la calidad de las respuestas."
RAG Token Embedding
Context window 🤖 LLM
El tamaño máximo de texto que el LLM puede "ver" en una sola petición.
+
En una frase
Es la "memoria de trabajo" del LLM por turno. GPT-4o tiene ~128k tokens; Claude Sonnet ~200k; Gemini hasta 1M. Aún así nunca alcanza para una empresa entera.
Por qué importa
Llenar el context window cada vez es caro y lento. RAG solo trae los pedazos relevantes — ahí el ahorro.
"¿1 millón de tokens? Bonito en demo. En producción, ¿van a pagar 1M de tokens por cada pregunta? Por eso existe RAG."
Token RAG
Cosine similarity 🧠 vectores
La "regla" matemática para medir qué tan parecidos son dos vectores.
+
En una frase
Mide el ángulo entre dos vectores (no la distancia). 1 = misma dirección, 0 = perpendicular, -1 = opuestos. Es lo que Vector Search usa por defecto.
Variantes
Otras métricas: dotProduct (más rápido, requiere vectores normalizados), euclidean (distancia recta). En MongoDB lo eliges en el índice.
"Cuando alguien pregunta 'cómo decide MongoDB qué documentos son similares' — es cosine similarity. Misma matemática que usa Spotify para recomendarte música."
Vector Embedding
Dimensiones 🧠 vectores
Cuántos números tiene un vector. Típico: 1024 o 1536. Más = más matiz, más caro.
+
En una frase
Cada dimensión es un eje que captura algún aspecto del significado. 1536 dimensiones = 1536 números que juntos describen la pieza de texto en el espacio semántico.
El trade-off
Más dimensiones = más matiz, pero también más RAM, más cómputo, más storage. Voyage usa Matryoshka — puedes truncar el vector y seguir teniendo buena precisión.
"Cuando hablen de dimensiones: ese número define cuánto espacio y RAM va a comer su solución. Es la primera variable de costo. Voyage AI optimiza esto."
Vector Embedding Voyage
Embedding 🧠 vectores
El proceso de convertir texto (o imagen, audio) en un vector que captura su significado.
+
En una frase
Un modelo de IA "lee" un pedazo de contenido y produce una lista de números (vector) que representa su significado en un espacio matemático. Cosas similares = vectores cercanos.
El truco
"Rey" - "Hombre" + "Mujer" ≈ "Reina". La aritmética funciona porque el modelo capturó relaciones reales del lenguaje.
"Embedding es como una huella digital del significado. Dos textos con la misma idea tienen huellas parecidas — aunque usen palabras distintas."
Vector Cosine Voyage AI Dimensiones
Modelo de embedding 🧠 vectores
El cerebro que produce embeddings. Voyage, OpenAI, Cohere — cada uno con su sabor.
+
En una frase
Una red neuronal entrenada específicamente para producir embeddings de calidad. NO confundir con un LLM — un embedding model no genera texto, solo vectores.
Cómo elegir
Para texto general: voyage-4. Código: voyage-code-3. Legal: voyage-law-2. Finanzas: voyage-finance-2. Multimodal (texto + imagen + video): voyage-multimodal-3.5.
"¿Su modelo de embedding actual los está limitando? Migrarse cuesta porque hay que re-vectorizar todo. Por eso elegir bien desde el día uno importa."
Voyage AI Embedding
HNSW 🔎 búsqueda
El tipo de índice que usa Vector Search por debajo. "Hierarchical Navigable Small World".
+
En una frase
Una estructura tipo "grafo de vecinos" que permite encontrar vectores similares en milisegundos sin comparar contra todos. Sin un índice así, buscar entre 1M de vectores tardaría minutos.
Por qué te importa
Es lo que hace pgvector lento a millones de vectores: el HNSW pesa en RAM y los builds tardan horas. Es resoluble con tuning, pero ¿quieren operar eso o construir su producto?
"Si te preguntan tecnicismos: HNSW es el algoritmo. MongoDB lo gestiona automáticamente. En Pinecone o pgvector tienes que tunearlo a mano."
Vector Search Atlas Vector Search
Hybrid Search 🔎 búsqueda
Combinar full-text + vector search en una sola consulta. Lo mejor de los dos mundos.
+
En una frase
Corres ambas búsquedas en paralelo (keywords + semántica), combinas los resultados con $rankFusion. Cubre exactitud Y comprensión.
Caso real
Kovai.co lo usa para su chatbot Eddy AI: +30% en precisión de retrieval al usar Hybrid Search nativo de Atlas.
"En cualquier otra arquitectura tienen que orquestar Elastic + Pinecone + lógica de fusión. En MongoDB es un solo aggregation pipeline."

LLM 🤖 LLM
Large Language Model. Predice la siguiente palabra. Eso es todo. Lo demás es cómo lo usas.
+
En una frase
Una red neuronal gigante (cientos de billones de parámetros) que aprendió a predecir la siguiente palabra de una secuencia. Ejemplos: GPT-4o, Claude Sonnet, Gemini, Llama.
Lo crítico
El LLM no sabe nada de tu cliente. Solo sabe lo que vio en su entrenamiento. Por eso necesita RAG para tener contexto del negocio.
"El LLM es el cerebro. Pero un cerebro sin información de tu negocio no sirve. Ahí entran los datos — que viven en MongoDB."

MCP · Model Context Protocol 🚀 agentes
El "USB universal" para conectar agentes a herramientas. Estándar de Anthropic (nov 2024).
+
En una frase
Un protocolo abierto que define cómo un agente se conecta a fuentes de datos y herramientas externas. Antes de MCP, cada integración era a mano. Con MCP es plug-and-play.
Por qué te importa
MongoDB tiene su propio MongoDB MCP Server oficial (anunciado oct 2025). Cualquier agente — Claude, OpenAI, Microsoft Foundry — puede conectarse a Atlas en minutos.
"MCP es como cuando estandarizaron el USB. Antes cada cable era único. Ahora cualquier agente conecta a Mongo sin escribir integración custom."
Agente MongoDB MCP Server
MongoDB MCP Server 🍃 MongoDB AI
El conector oficial que permite a agentes (Claude, GPT, etc.) explorar y operar Atlas.
+
En una frase
Un servidor MCP que expone tu base MongoDB (Atlas, Community, Enterprise) a cualquier agente que hable MCP: Claude Code, Cursor, Windsurf, Microsoft Foundry.
Qué hace
El agente puede: listar colecciones, ver schemas, ejecutar queries, crear clusters, gestionar usuarios — todo desde lenguaje natural con compliance enterprise.
Caso real
Anuncio en Microsoft Ignite 2025: MongoDB MCP Server en el Tool Catalog de Microsoft Foundry. Es el primer connector enterprise en esa plataforma.
"Es la diferencia entre 'integramos con IA' y 'tu IA habla con nuestros datos nativamente'. Cero código glue."

RAG · Retrieval-Augmented Generation 📚 RAG
"Búscalo antes de responder." El patrón que cura las alucinaciones del LLM.
+
En una frase
Retrieval (recuperas docs reales con Vector Search) → Augmented (los inyectas como contexto) → Generation (el LLM responde basado en ellos).
Por qué es enorme
Cerca de 7 de cada 10 empresas que usan IA generativa ya implementan RAG (Snowflake, Databricks). Es EL caso de uso dominante en producción.
"La pregunta nunca es si tu cliente va a hacer RAG — es dónde va a guardar los vectores. Y ahí cerramos."
Vector Search Chunking Embedding Reranker
$rankFusion 🔎 búsqueda
El operador de aggregation que combina resultados de múltiples búsquedas. Disponible desde Atlas 8.1.
+
En una frase
Una etapa del aggregation pipeline que toma resultados de N búsquedas (full-text, vector, geo, lo que sea) y los fusiona en un ranking unificado usando Reciprocal Rank Fusion.
El truco matemático
Cada doc obtiene un score = 1 / (rank + 60) en cada pipeline. Los scores se suman ponderados. Sin matemáticas que tunear: funciona out-of-the-box.
"Es UN operador. $rankFusion. En Pinecone + Elastic tendrían que escribir lógica de fusión en código. Aquí: una línea del pipeline."

ReAct · Reason + Act 🚀 agentes
El patrón principal de cómo piensa un agente: razona, actúa, observa, repite.
+
En una frase
Un loop documentado en un paper de 2023: el agente pone en palabras su razonamiento ("necesito buscar el calendario") antes de cada acción ("call calendar_api"). Eso le da transparencia y permite corregirse.
Por qué importa
Es el patrón que hace que un agente sea auditable. Para regulados (banca, salud) es no-negociable: hay que poder explicar cada decisión que tomó.
"Para regulados: ReAct deja un log de razonamiento explícito. Cada acción del agente está justificada. Eso es lo que pide compliance."

Reranker 📚 RAG
Modelo que reordena los resultados de búsqueda por relevancia real, después del retrieval.
+
En una frase
Vector Search te trae los top 100 candidatos rápido. Un reranker (más lento pero más preciso) los reordena para que el top 10 sea realmente lo mejor.
Voyage Rerankers
Voyage AI ofrece rerank-2 y rerank-2-lite — diseñados para optimizar relevancia manteniendo latencia bajo control. Mejoran la precisión sin re-vectorizar todo.
"Si su RAG tiene precisión baja, no es problema del LLM — es del retrieval. Un reranker arregla eso sin cambiar nada más."

Búsqueda semántica 🔎 búsqueda
Buscar por significado, no por palabras exactas. Sinónimo casi-exacto de Vector Search.
+
En una frase
Buscas "solución de enfriamiento" y encuentra "aire acondicionado" — aunque las palabras no coincidan. Funciona porque los embeddings capturaron que ambos significan lo mismo.
"Es lo opuesto a la búsqueda de Google de hace 15 años. Antes buscabas keywords. Hoy buscas ideas. Eso es búsqueda semántica."

Temperatura 🤖 LLM
Cuán creativo vs determinista responde el LLM. 0 = robot. 1+ = poeta.
+
En una frase
Parámetro de 0 a 2.0 que controla cuánto "se arriesga" el LLM al elegir la siguiente palabra. Para RAG en producción: 0 o 0.1 (queremos respuestas consistentes).
"Si su cliente está usando temperatura alta en banca, es bandera roja. Producción seria = temperatura baja."

Token 🤖 LLM
La unidad mínima que procesa un LLM. ≈ ¾ de palabra en inglés, ½ en español.
+
Por qué importa
Los APIs cobran por token. El español pesa más que el inglés. Si tu cliente procesa muchos documentos en español, costo previsto se duplica.
"Cada vez que pregunten cuánto cuesta — primer paso es contar tokens, no palabras. Y para LATAM, eso es 30-50% más caro que el demo en inglés."

Tool use 🚀 agentes
Cuando el LLM llama a una función externa (API, base, calculadora) en lugar de responder con texto.
+
En una frase
El LLM puede emitir un "llamado a función" estructurado en JSON. Tu código ejecuta esa función real y le devuelve el resultado al LLM. Es lo que convierte a un chatbot en agente.
Sin tool use
El LLM responde "creo que el precio es ~$50". Con tool use: el LLM llama get_price(item), obtiene $47.32, responde con el dato real.
"Tool use es lo que separa los demos de los productos. Sin esto, el LLM solo opina. Con esto, ejecuta."

Vector 🧠 vectores
Una lista de números (típico 1024-1536). El resultado de un embedding.
+
En una frase
[0.23, -0.81, 0.45, 0.12, -0.67, ...] — un vector es solo esa lista. Lo importante es que dos vectores que estén "cerca" en ese espacio matemático representan significados cercanos.
"Para el cliente: piensen en un vector como coordenadas en un mapa. Cosas parecidas viven cerca. Cosas distintas, lejos."

Vector Search 🔎 búsqueda
Buscar entre vectores para encontrar los más cercanos a una query. La pieza fundamental del RAG.
+
En una frase
Vectorizas la pregunta, vectorizas los docs, encuentras los docs cuyos vectores son matemáticamente más parecidos al vector de la pregunta. Eso es Vector Search.
Implementaciones
MongoDB: $vectorSearch en aggregation. Pinecone: API dedicada. pgvector: extensión de Postgres. Cada una con su trade-off de operación.
"Vector Search es la categoría. Atlas Vector Search es nuestro producto. La diferencia: el nuestro vive en la misma base que sus datos operacionales."
