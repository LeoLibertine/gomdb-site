API que permite consultar información de los registros insertados en BD

Análisis de los Stored Procedures de consulta del sistema BDTL y su transformación a una arquitectura de microservicios optimizada para MongoDB.

📍 Contexto de Migración de APIs de Consulta

Los Stored Procedures de consulta representan las operaciones más críticas y frecuentes del sistema BDTL. Su migración a microservicios con MongoDB eliminará la complejidad de JOINs y optimizará dramáticamente el rendimiento de las consultas.

🔍

spGetDataSubTypeDocumentByDateAndHours

Consulta por Proceso, Subtipo y Rango de Fechas

Este SP está diseñado para una consulta específica: buscar registros por Proceso, Subtipo Documental y un rango de fecha/hora. Su lógica se puede descomponer en cuatro pasos clave que serán simplificados radicalmente en la nueva arquitectura.

1

Preparación de Parámetros

Recibe los filtros a través de un tipo de tabla (@Parameters) y los convierte en variables de fecha (@FechaInicio, @FechaFin) y paginación (@PageSize, @Page).

Migración: Los parámetros se reciben directamente como JSON en el microservicio
2

Consulta y JOIN

Ejecuta la consulta principal con la operación más costosa: filtra DatosLiquidos por Proceso y TipoDocumento, realiza un LEFT OUTER JOIN con API.Estados para obtener la descripción del estado del documento.

Eliminado: El JOIN desaparece gracias a la desnormalización
3

Uso de Tabla Temporal

Almacena todos los resultados de la consulta en una tabla temporal (#TMPCONSULTARPORFECHA). Esto consume recursos y añade un paso intermedio innecesario.

Eliminado: La consulta se ejecuta directamente sobre la colección
4

Paginación con SQL Dinámico

Para poder paginar los resultados, el SP construye una consulta como un string de texto (@SQL) y la ejecuta con sp_executesql. Técnica compleja y propensa a errores.

Reemplazado: Por .skip() y .limit() nativos de MongoDB

⚖️ Comparación: SQL Server vs MongoDB

❌ SQL Server (Complejo)
-- Paso 1: Preparar parámetros
DECLARE @FechaInicio DATETIME
DECLARE @FechaFin DATETIME
DECLARE @PageSize INT

-- Paso 2: JOIN complejo
SELECT DL.*, E.Descripcion
FROM DatosLiquidos DL
LEFT OUTER JOIN API.Estados E
ON DL.IdEstadoDocumento = E.Id
WHERE DL.Proceso = @Proceso

-- Paso 3: Tabla temporal
INTO #TMPCONSULTARPORFECHA

-- Paso 4: SQL dinámico
SET @SQL = 'SELECT * FROM #TMP...'
EXEC sp_executesql @SQL
VS
✅ MongoDB (Simple)
// Parámetros directos del request
const
proceso = "EstudioCreditoPyPConsumo";
const
subTipo = "Anexo de riesgo";
const
fechaInicio = new Date("2019-01-14T08:00:00Z");
// Una sola consulta sin JOINs
const
resultados = await db.collection('registros_bdtl')
.find({
"Proceso": proceso,
"TipoDocumento": subTipo,
"FechaKofax": { $gte: fechaInicio }
})
.sort({ "FechaKofax": 1 })
.skip((pagina - 1) * tamanoPagina)
.limit(tamanoPagina)
.toArray();

spObtenerDatosLiquidosClienteFechas COMPLEJO

Búsqueda dentro de campos JSON

Este SP es un ejemplo aún más potente de cómo la migración a MongoDB simplificará radicalmente la lógica. La característica más importante es que necesita buscar valores dentro del campo JSON, lo cual es una operación muy compleja y lenta en SQL Server, pero es una de las fortalezas nativas de MongoDB.

1

Parámetros Complejos

Acepta un tipo de tabla (@ParametrosJSON) que le permite recibir una lista de pares clave/valor para buscar dentro del campo DatosLiquidos.

2

Lógica Delegada

La parte más difícil no la hace este SP directamente. Construye una cadena de filtros (@CADENABUSQPRINCIPAL) y llama a otro SP (API.spBusqJsonDatosLiquidos) para hacer el trabajo pesado.

3

Tabla Temporal

Depende de una tabla temporal (#TMPCONSULTAAPI) para almacenar los resultados intermedios antes de poder paginarlos.

4

SQL Dinámico

Utiliza SQL dinámico para aplicar el OFFSET y FETCH, una técnica que puede ser ineficiente y difícil de mantener.

🎯 La Funcionalidad Más Crítica

La funcionalidad clave y más costosa es la búsqueda dentro de un campo de texto que contiene JSON. En SQL Server esto requiere múltiples SPs, lógica compleja y operaciones costosas.

🔍 Ejemplo de Búsqueda:

Buscar todos los documentos donde DatosLiquidos contenga "TipoDocCliente": "CC" y "NumeroDocCliente": "1016041796"

💡 En MongoDB: Esta operación compleja se convierte en una consulta nativa usando "dot notation"

🔄 Transformación Completa a MongoDB

Comparación de la Lógica Completa

1. Búsqueda dentro del JSON
Reemplazado por Dot Notation: MongoDB permite consultar campos dentro de un objeto JSON de forma nativa usando "DatosLiquidos.NumeroDocCliente". Con índices, la búsqueda es extremadamente rápida.
2. Lógica Delegada
Eliminado: No hay necesidad de delegar la lógica. La consulta completa se puede expresar en una única operación find() en el microservicio.
3. Uso de Tabla Temporal
Eliminado: La consulta se ejecuta directamente sobre la colección registros_bdtl.
4. Paginación con SQL Dinámico
Reemplazado: Por .skip() y .limit() nativos de MongoDB.

💻 Ejemplo: La Nueva Consulta en MongoDB

✨ Consulta Completa en MongoDB (Microservicio)
// Parámetros recibidos por el microservicio
const tipoDocumento = "Anexo de riesgo";
const fechaInicio = new Date("2018-01-01T00:00:00Z");
const fechaFin = new Date("2018-11-14T23:59:59Z");
const pagina = 1;
const tamanoPagina = 15;

// Parámetros JSON que ahora son parte del filtro principal
const filtroJson = {
"DatosLiquidos.TipoDocCliente": "CC",
"DatosLiquidos.NumeroDocCliente": "1016041796"
};

// La consulta completa en MongoDB
const resultados = await db.collection('registros_bdtl').find({
// --- Filtros ---
"TipoDocumento": tipoDocumento,
"FechaKofax": {
$gte: fechaInicio,
$lte: fechaFin
},
...filtroJson // Se añaden dinámicamente los filtros del JSON
})
.sort({ "FechaKofax": 1, "IdDocumento": 1 })
.skip((pagina - 1) * tamanoPagina)
.limit(tamanoPagina)
.toArray();

// Conteo total para la paginación
const totalRegistros = await db.collection('registros_bdtl')
.countDocuments({
"TipoDocumento": tipoDocumento,
"FechaKofax": { $gte: fechaInicio, $lte: fechaFin },
...filtroJson
});

💡 Nota sobre el Código

Este ejemplo muestra cómo un Stored Procedure complejo de más de 80 líneas se reduce a una consulta MongoDB elegante y eficiente. El código es completamente funcional y listo para implementar en un microservicio AWS Lambda.

🎯 La Demostración Perfecta de los Beneficios

Este caso de uso es la demostración perfecta de los beneficios de la migración. Un proceso que en SQL Server requiere múltiples SPs, tablas temporales y lógica compleja para simular una búsqueda dentro de un JSON, en MongoDB se convierte en una consulta nativa, simple, legible y de alto rendimiento.

API.spGetDatosLiquidosJson MUY COMPLEJO

Constructor de Consultas Dinámicas Avanzado

Este SP es un constructor de consultas dinámicas muy avanzado. Su propósito es ofrecer un único punto de entrada para realizar búsquedas complejas con múltiples filtros opcionales. Es probablemente el ejemplo más claro de por qué se necesita la migración a MongoDB.

1

Validación y Preparación

Valida que los campos obligatorios no estén vacíos. Luego, construye dinámicamente las cláusulas WHERE consultando una tabla de parámetros ([API].[Parameters]) para mapear los nombres de los filtros a los nombres reales de las columnas.

2

Consulta SQL Dinámica

Construye una consulta SQL como un string de texto para obtener un conjunto inicial de datos basado en los filtros de metadatos (Entidad, Proceso, Fechas, etc.).

3

El Bucle para "Parsear" el JSON

La parte más crítica y costosa. Para filtrar por un campo dentro del DatosLiquidos, utiliza un bucle WHILE y la función OPENJSON para recorrer de forma recursiva la estructura del JSON de cada registro. Proceso extremadamente lento e intensivo en CPU.

Crítico: "Desarmando" el JSON en SQL nivel por nivel
4

Paginación y JOIN Final

Después del complejo proceso de filtrado de JSON, utiliza tablas temporales y ROW_NUMBER() para paginar los resultados y finalmente hace un JOIN con la tabla Estados.

🔄 Transformación del SP Más Complejo

Comparación de Lógica - spGetDatosLiquidosJson

1. Construcción de SQL Dinámico
Reemplazado por Construcción de Objeto de Filtro: El microservicio construye un objeto JSON de filtro de forma programática, añadiendo solo las claves que el usuario proporcionó. Esto es más seguro y mucho más simple.
2. Bucle WHILE y OPENJSON
Reemplazado por Dot Notation: Esta es la mayor ganancia. La búsqueda dentro del JSON se convierte en una parte nativa del filtro. Para buscar un ValorDatoLiquido en un CampoDatoLiquido, la sintaxis es directa y no requiere ningún bucle.
3. Múltiples Tablas Temporales
Eliminado: La consulta se ejecuta directamente sobre la colección y se pagina con los métodos nativos .skip() y .limit().
✨ Ejemplo: Búsqueda de Placa en MongoDB
// Parámetros recibidos por el microservicio
const filtros = {
"Proceso": "Desembolsos",
"TipoDocumento": "Cedula de ciudadania",
"FechaKofax": {
"$gte": new Date("2024-06-01T00:00:00Z"),
"$lte": new Date("2024-06-06T23:59:59Z")
},
// --- Búsqueda dentro del JSON ---
// La clave se construye dinámicamente
"DatosLiquidos.placa": "HJG333"
};

const pagina = 1;
const tamanoPagina = 50;

// La consulta completa en MongoDB
const resultados = await db.collection('registros_bdtl').find(filtros)
.sort({ "IdDocumento": 1 })
.skip((pagina - 1) * tamanoPagina)
.limit(tamanoPagina)
.toArray();

// Conteo para la paginación
const totalRegistros = await db.collection('registros_bdtl')
.countDocuments(filtros);

🎯 Conclusión Crítica

Este SP es el ejemplo perfecto de cómo se fuerza a una base de datos relacional a comportarse como una base de datos de documentos. Al migrar a MongoDB, reemplazamos un código procedural, frágil y lento por una consulta declarativa, simple y extremadamente rápida, especialmente si se crea un índice en el campo DatosLiquidos.placa.

API.spGetDatosLiquidosProcesoFechas

Constructor de Consultas Dinámicas - Motor de Búsqueda

Este SP es un excelente ejemplo de un "constructor de consultas dinámicas" en SQL. Su propósito es ser un punto de entrada flexible para buscar registros con múltiples filtros opcionales. Es esencialmente un motor de búsqueda para los metadatos de los registros.

1

Construcción de SQL Dinámico

El núcleo del SP es la construcción de un string de texto (@SQL) que se convierte en la consulta final. Utiliza varias técnicas para manejar los filtros opcionales.

2

Consulta a Tabla de Parámetros

Busca en la tabla [API].[Parameters] para traducir los nombres de los filtros que llegan de la API a los nombres de las columnas de la base de datos.

3

Subconsulta Ineficiente

Para obtener la descripción del estado, ejecuta una subconsulta (select Description from API.Estados ...) por cada fila del resultado. Esto es funcionalmente un JOIN y puede ser muy lento en conjuntos de datos grandes.

Problema: Una subconsulta por cada fila
4

Paginación con Funciones de Ventana

Utiliza COUNT(*) OVER() para calcular el total de filas y OFFSET ... FETCH para la paginación. Aunque es una técnica moderna en SQL, sigue siendo parte de una consulta dinámica compleja.

🔄 Transformación del Motor de Búsqueda

Comparación de Lógica - spGetDatosLiquidosProcesoFechas

1. Construcción de SQL Dinámico
Reemplazado por Construcción de Objeto de Filtro: El microservicio crea un objeto de filtro y solo añade las claves correspondientes a los parámetros que el usuario envió. Es más simple, seguro y no requiere concatenar strings.
2. Subconsulta para el Estado
Eliminado: Gracias al modelo desnormalizado, el documento IdEstadoDocumento con su descripción ya está incrustado en el registro principal. No se necesita ninguna operación adicional para obtenerlo.
3. Paginación con Funciones de Ventana
Reemplazado por .skip() y .limit(): Se usan los métodos nativos de MongoDB para una paginación simple y eficiente.
✨ Ejemplo: Motor de Búsqueda en MongoDB
// Parámetros recibidos por el microservicio
const pagina = 1;
const tamanoPagina = 10;

// El microservicio construye el objeto de filtro dinámicamente
const filtro = {
"Entidad.valor": "Bancolombia",
"Proceso": "MiValorProceso", // Asumiendo que se busca por nombre
"TipoDocumento": "MiTipoDocumento",
"FechaKofax": { // Asumiendo que se busca por FechaKofax
"$gte": new Date("2025-07-22T08:00:00Z"),
"$lte": new Date("2025-07-22T17:00:00Z")
}
};

// Se añaden los filtros opcionales si existen
if (canalDocumento) {
filtro["CanalDocumento.valor"] = canalDocumento;
}
if (tecnologiaExtraccion) {
filtro["TecnologiaExtraccion.valor"] = tecnologiaExtraccion;
}
// etc.

// La consulta completa en MongoDB
const resultados = await db.collection('registros_bdtl').find(filtro)
.sort({ "IdDocumento": 1 })
.skip((pagina - 1) * tamanoPagina)
.limit(tamanoPagina)
.toArray();

// Conteo para la paginación
const totalRegistros = await db.collection('registros_bdtl')
.countDocuments(filtro);

API.spBusqJsonDatosLiquidos CRÍTICO

Motor de Búsqueda JSON - El SP Más Importante

Este es probablemente el SP más importante de toda la lógica de consulta, ya que es el "motor" que los otros SPs utilizan para realizar la tarea más difícil en SQL Server: buscar valores dentro del campo JSON DatosLiquidos. Su análisis revela de la manera más clara los beneficios de la migración a MongoDB.

1

Seguridad y Validación

Realiza una limpieza básica de los parámetros para prevenir inyecciones de SQL. Luego, llama a otro SP (spValidarBusquedaDocCliente) para ejecutar reglas de negocio y validar que los parámetros de búsqueda sean correctos.

2

Construcción Dinámica de WHERE

Su función principal. Utiliza la función JSON_VALUE de SQL Server para acceder a los datos dentro del campo DatosLiquidos. Consulta tablas de mapeo (API.MapeoParametros, API.MapeoValoresTipoDocCliente) para traducir los nombres de los parámetros de la API a la ruta real dentro del JSON.

Ejemplo: NumeroDocCliente → $.client.id
3

Concatenación de Filtros

Concatena todos los filtros JSON en un único string de texto (@CADENABUSQJSON). Este string se convierte en la cláusula WHERE que se inyectará en la consulta principal.

4

Ejecución de SQL Dinámico

Toma el string de la cláusula WHERE construido, lo inyecta en una plantilla de INSERT ... SELECT y lo ejecuta usando sp_executesql. El resultado se inserta en una tabla temporal (#TMPCONSULTAAPI).

Crítico: Inyección de SQL construido dinámicamente

🎯 La Prueba Definitiva de la Migración

La existencia de este SP es la prueba definitiva de que se está forzando a una base de datos relacional a hacer algo para lo que no fue diseñada. En MongoDB, toda la funcionalidad de este SP es nativa, simple y no requiere un SP intermediario.
