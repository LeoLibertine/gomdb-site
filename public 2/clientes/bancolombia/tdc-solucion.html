Sharding en Atlas: Del Particionamiento Manual a la Automatización Total

La solución definitiva para gestionar miles de millones de registros sin la complejidad operativa del particionamiento manual

Su Enfoque Actual en PostgreSQL

  • × Particionamiento Manual: Crear particiones por mes manualmente
  • × Mantenimiento Complejo: Gestionar índices para cada partición
  • × Lógica en Aplicación: Código complejo para consultar múltiples particiones
  • × Escalabilidad Limitada: Difícil agregar más capacidad sin downtime

Sharding Automatizado en Atlas

  • ✓ 100% Automatizado: Atlas gestiona toda la complejidad
  • ✓ Sin Mantenimiento: Balanceo y optimización automáticos
  • ✓ Transparente: Una sola conexión, sin lógica compleja
  • ✓ Auto-Scaling: Crece automáticamente según demanda

Implementación "Push-Button" en 3 Pasos

1

Crear Cluster Sharded

Un click en la consola Atlas. La infraestructura compleja se provisiona automáticamente.

2

Definir Shard Key

Elegir la estrategia de distribución óptima para sus datos.

sh.shardCollection("tdc.transacciones", { fecha: 1, cuenta_id: 1 } )
3

¡Listo!

Atlas se encarga del resto. Balanceo, routing y escalado totalmente automáticos.

Auto-Scaling Inteligente

Modelado de Datos Inteligente

De Complejidad a Simplicidad

Ejemplo Simple: Consultar Cliente y Sus Tarjetas

❌ SQL - 2 Tablas + JOIN
SELECT c.nombre, t.numero, t.tipo FROM clientes c JOIN tarjetas t ON c.id = t.cliente_id WHERE c.id = 12345;

2 consultas a disco, JOIN costoso

✅ MongoDB - 1 Documento
db.clientes.findOne({ _id: 12345 }) // Resultado inmediato: { nombre: "María García", tarjetas: [ { numero: "****1234", tipo: "débito" }, { numero: "****5678", tipo: "crédito" } ] }

1 consulta, toda la info junta

❌ SQL Actual - Múltiples JOINs

SELECT t.transaction_id, t.amount, c.customer_name, a.account_number, ca.card_number, m.merchant_name FROM transactions t JOIN accounts a ON t.account_id = a.id JOIN customers c ON a.customer_id = c.id JOIN cards ca ON t.card_id = ca.id JOIN merchants m ON t.merchant_id = m.id WHERE t.date >= '2024-01-01' AND t.amount > 500000 ORDER BY t.date DESC;

✅ MongoDB - Documento Único

db.transactions.find({ date: { $gte: ISODate("2024-01-01") }, amount: { $gt: 500000 } }).sort({ date: -1 }) // Toda la información ya está embebida: { transaction_id: "TXN123", amount: 750000, customer: { name: "Juan Pérez" }, account: { number: "***4567" }, card: { number: "****1234" }, merchant: { name: "Comercio XYZ" } }

Catálogo de Patrones de Modelado Esenciales

🏠 Básico Patrón de Embebido (Embedding)

Anidar documentos relacionados dentro de un documento principal

// Cliente con sus direcciones embebidas { _id: "CLI123", nombre: "María García", direcciones: [ { tipo: "casa", calle: "Cra 7 # 123-45", ciudad: "Bogotá" }, { tipo: "oficina", calle: "Calle 100 # 15-20", ciudad: "Bogotá" } ] }

💡 Cuándo usarlo:

Para relaciones "uno a pocos" (cliente con 2-3 direcciones)

✅ Beneficio:

Elimina JOINs. Todos los datos en una sola operación de lectura

🔗 Básico Patrón de Referencia (Referencing)

Almacenar el _id de un documento en otro, similar a clave foránea

// Cliente (colección: clientes) { _id: "CLI456", nombre: "Juan Pérez" } // Transacciones (colección: transacciones) { _id: "TXN789", cliente_id: "CLI456", // Referencia monto: 150000, fecha: ISODate("2024-01-15") }

💡 Cuándo usarlo:

Para relaciones "uno a muchos" (cliente con miles de transacciones)

✅ Beneficio:

Evita duplicación masiva y documentos que excedan 16MB

⭐ Optimización Clave Patrón de Subconjunto (Subset Pattern)

Duplicar solo la información más consultada para evitar lookups

// En vez de hacer $lookup para el nombre: { transaction_id: "TXN123", monto: 250000, // Subconjunto del cliente embebido: cliente: { id: "CLI789", nombre: "Ana López", // Solo campos tipo: "Premium" // necesarios } // NO embebemos todo el documento del cliente }

💡 Ejemplo Práctico:

Lista de transacciones mostrando nombre del cliente sin cargar todo su perfil

✅ Beneficio:

Optimiza drásticamente las lecturas más comunes. No más $lookup para datos básicos

🚀 Alto Volumen Patrón Cubo (Bucket Pattern)

Agrupar cientos de eventos en un único documento por período

// En vez de 1 doc por transacción: { cuenta_id: "ACC123", fecha: ISODate("2024-01-15"), contador_transacciones: 156, monto_total: 15750000, transacciones: [ { timestamp: ISODate("2024-01-15T09:15:00"), monto: 50000, tipo: "compra", comercio: "Almacén XYZ" }, // ... hasta 500 transacciones del día ] }

💡 Implementación:

Un documento por cuenta por día, conteniendo todas las transacciones

✅ Beneficio:

Reduce documentos e índices en órdenes de magnitud. Consultas por fecha ultra-rápidas

🎯 Guía Estratégica: ¿Cómo Elegir el Patrón Correcto?

Si la relación es... Y los datos se leen... Y el tamaño/volatilidad es... El patrón recomendado es...
Uno a pocos
(cliente-direcciones)		 Casi siempre juntos Pequeño y estable Embebido
Uno a muchos
(cliente-transacciones)		 A menudo por separado Grande o volátil Referencia
Uno a "muchísimos"
(logs, IoT, transacciones)		 Por rangos de tiempo Pequeño por evento Cubo (Bucket)
Cualquiera Necesito info básica rápido - Subconjunto

Regla de Oro: Modelamos los datos según cómo la aplicación los necesita, no según relaciones teóricas
